# Aheart UI Full Component Roadmap and I18n Design

## Goal

Turn Aheart UI from a single ready component into a product-grade Vue 3 component library with an Ant Design-style documentation website, Chinese as the default documentation language, English under `/en/`, and a phased component roadmap that never presents unfinished components as complete.

## Current State

- The package currently exports `Button` as the only ready source component.
- The documentation site already has an Ant Design-style shell, homepage, guide pages, component overview, and Button page.
- The component overview marks most components as `Planned`; this is accurate and should remain accurate until each component has source, styles, tests, exports, and docs.
- `docs/.vitepress/cache/` is generated by the local docs server and must stay out of commits.

## Product Principles

- Default to Chinese for the root site because the primary user expectation is Chinese documentation.
- Provide a mirrored English site at `/en/` so the package can look credible for npm, GitHub, and future external users.
- Prefer small, stable, composable components before complex data components.
- Do not clone Ant Design API surface blindly. Use Ant Design as the completeness benchmark, but keep Aheart UI APIs Vue-friendly, typed, and maintainable for this codebase.
- A component becomes `Ready` only when implementation, exports, tests, examples, docs, and theme tokens are all present.

## Documentation I18n Design

Use VitePress locales:

- Root locale `/`: Simplified Chinese, default language.
- English locale `/en/`: English mirror with equivalent navigation, sidebar, guide pages, component overview, and component pages.
- Keep shared visual theme files in `docs/.vitepress/theme/`.
- Keep component metadata language-aware instead of duplicating status logic. The metadata should expose Chinese and English labels while sharing stable component keys, statuses, and links.

Recommended metadata shape:

```ts
export type Locale = 'zh' | 'en'
export type ComponentStatus = 'Ready' | 'Planned'

export interface ComponentMeta {
  key: string
  name: string
  zhName?: string
  description: Record<Locale, string>
  status: ComponentStatus
  link: Record<Locale, string>
}
```

This keeps release status consistent across languages while allowing each locale to render natural copy.

## Runtime I18n Scope

Phase 1 covers documentation i18n only. Runtime component locale support should be introduced with a future `ConfigProvider` when components contain built-in human text, such as `Empty`, `Pagination`, `Table`, `Modal`, `Upload`, or date/time pickers.

The future runtime locale layer should include:

- `ConfigProvider` for `locale`, theme tokens, and global component defaults.
- Locale packs such as `zhCN` and `enUS`.
- Component-level fallback to Chinese defaults when no provider exists.

## Component Delivery Standard

Every component should ship with:

- `packages/components/src/<component>/<component>.vue`
- `packages/components/src/<component>/types.ts`
- `packages/components/src/<component>/style.css`
- `packages/components/src/<component>/index.ts`
- focused tests under `packages/components/src/<component>/__tests__/`
- root export from `packages/components/src/index.ts`
- plugin installation support through the existing `withInstall` pattern
- Chinese docs under `docs/components/<component>.md`
- English docs under `docs/en/components/<component>.md`
- component overview metadata updated from `Planned` to `Ready`
- examples covering common states, disabled/loading/size variants where relevant, and accessibility expectations

## Phased Roadmap

### Phase 1: I18n Foundation and Lightweight Ready Components

Purpose: make the site bilingual and expand beyond Button with low-risk building blocks.

Deliver:

- VitePress locale config with Chinese root and English `/en/`.
- Chinese navigation/sidebar text fixed at the source encoding level.
- English guide pages and component overview.
- Language-aware component metadata.
- Ready components: `Icon`, `Space`, `Divider`, `Tag`, `Alert`.

Why these first:

- They are useful across docs examples and future components.
- They have limited state and limited cross-component dependency.
- They exercise the standard component folder, style, test, export, and docs workflow without taking on form or overlay complexity too early.

### Phase 2: General and Layout Completion

Deliver:

- `Typography`
- `Flex`
- `Grid`

Purpose: finish basic composition primitives for real product screens and richer documentation examples.

### Phase 3: Basic Data Entry

Deliver:

- `Input`
- `Textarea`
- `InputNumber`
- `Checkbox`
- `Radio`
- `Switch`

Purpose: establish controlled/uncontrolled value patterns, keyboard behavior, disabled states, focus states, and form-control styling.

### Phase 4: Select and Form System

Deliver:

- `Select`
- `Form`
- `FormItem`

Purpose: create the first real cross-component system, including validation messages, labels, help text, required markers, and consistent control sizing.

### Phase 5: Feedback and Overlay Layer

Deliver:

- `Message`
- `Modal`
- `Drawer`
- `Tooltip`
- `Popover`
- `Popconfirm`
- `Spin`
- `Skeleton`

Purpose: introduce portal/teleport behavior, focus management, z-index tokens, placement logic, global feedback APIs, and motion rules.

### Phase 6: Data Display

Deliver:

- `Badge`
- `Card`
- `Empty`
- `Descriptions`
- `Pagination`
- `Table`

Purpose: make the library useful for admin systems and information-heavy product UI. `Table` should come after simpler display components because it needs pagination, loading, empty state, sorting, layout, and accessibility decisions.

### Phase 7: Navigation and Advanced Components

Deliver:

- `Tabs`
- `Breadcrumb`
- `Dropdown`
- `Menu`
- `Steps`
- `DatePicker`
- `TimePicker`
- `Upload`
- `Tree`
- `TreeSelect`
- `Cascader`

Purpose: complete the Ant Design-style coverage benchmark after core patterns, overlays, data entry, and display foundations are stable.

## First Implementation Slice

The next implementation plan should cover Phase 1 only:

1. Add documentation i18n with Chinese root and English `/en/`.
2. Convert component metadata to language-aware status data.
3. Add `Icon`, `Space`, `Divider`, `Tag`, and `Alert` as ready components.
4. Add Chinese and English docs pages for all ready components.
5. Run and pass component tests, typecheck, package build, and docs build.
6. Commit and push the completed slice to GitHub.

This slice is large enough to prove the full production workflow and small enough to finish without mixing in unfinished form, overlay, or table architecture.

## Testing and Verification

For each implementation slice:

- `pnpm test`
- `pnpm typecheck`
- `pnpm build`
- `pnpm docs:build`

For docs UI changes, also run the docs dev server and inspect the homepage, guide pages, overview page, and changed component pages in desktop and mobile widths.

## Git and Release Practice

- Use a `codex/` branch for each meaningful slice.
- Keep generated cache and build output out of commits unless the package explicitly expects published build artifacts.
- Commit after each completed task group.
- Push the branch after each verified slice so GitHub has a recoverable checkpoint.
- npm publish and server deployment should happen only after package metadata, docs, build output, and credentials are confirmed by the user.

## Open Decisions Deferred

- Runtime `ConfigProvider` locale API is deferred until text-heavy components are implemented.
- Exact icon source is deferred to Phase 1 planning. The likely choices are a small internal SVG icon set or a peer dependency on a Vue icon package.
- Table virtualization, date library selection, upload transport API, and tree performance strategy are deferred to their own phase specs because they each deserve focused design.
